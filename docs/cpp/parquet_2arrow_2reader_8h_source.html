53 // Arrow read adapter class for deserializing Parquet files as Arrow row
54 // batches.
55 //
56 // This interfaces caters for different use cases and thus provides different
57 // interfaces. In its most simplistic form, we cater for a user that wants to
58 // read the whole Parquet at once with the FileReader::ReadTable method.
59 //
60 // More advanced users that also want to implement parallelism on top of each
61 // single Parquet files should do this on the RowGroup level. For this, they can
62 // call FileReader::RowGroup(i)->ReadTable to receive only the specified
63 // RowGroup as a table.
64 //
65 // In the most advanced situation, where a consumer wants to independently read
66 // RowGroups in parallel and consume each column individually, they can call
67 // FileReader::RowGroup(i)->Column(j)->Read and receive an arrow::Column
68 // instance.
69 //
70 // TODO(wesm): nested data does not always make sense with this user
71 // interface unless you are only reading a single leaf node from a branch of
72 // a table. For example:
73 //
74 // repeated group data {
75 // optional group record {
76 // optional int32 val1;
77 // optional byte_array val2;
78 // optional bool val3;
79 // }
80 // optional int32 val4;
81 // }
82 //
83 // In the Parquet file, there are 3 leaf nodes:
84 //
85 // * data.record.val1
86 // * data.record.val2
87 // * data.record.val3
88 // * data.val4
89 //
90 // When materializing this data in an Arrow array, we would have:
91 //
92 // data: list<struct<
93 // record: struct<
94 // val1: int32,
95 // val2: string (= list<uint8>),
96 // val3: bool,
97 // >,
98 // val4: int32
99 // >>
100 //
101 // However, in the Parquet format, each leaf node has its own repetition and
102 // definition levels describing the structure of the intermediate nodes in
103 // this array structure. Thus, we will need to scan the leaf data for a group
104 // of leaf nodes part of the same type tree to create a single result Arrow
105 // nested array structure.
106 //
107 // This is additionally complicated "chunky" repeated fields or very large byte
108 // arrays
